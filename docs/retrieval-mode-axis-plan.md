# Retrieval Mode Axis Implementation Plan

## Goal

Separate **when and why retrieval runs** from **how retrieval ranks memories**.

Existing and future baselines remain retrieval conditions:

- `no_memory`
- `input_rag`
- output-triggered RAG variants
- PARM

Memory-using conditions must independently select one of three retrieval modes:

- `dense`
- `hybrid`
- `enhanced`

These modes are not new baselines. Experiments should compare retrieval conditions under the same retrieval mode and compare retrieval modes within the same condition.

## Ownership Boundary

GBrain is responsible for preparing the memory substrate:

- importing and chunking pages
- embedding chunks with `openai:text-embedding-3-small` at 512 dimensions
- retaining page titles and source metadata
- extracting page-to-page graph edges

GBrain must not perform ranking during canonical benchmark runs. A tracked exporter will freeze its neutral index data into a versioned artifact. PARM will own BM25, cosine ranking, RRF, title retrieval, graph reranking, fixed top-k selection, and retrieval traces.

Canonical runs must never invoke `gbrain search`.

## Frozen Retrieval Artifact

Add a tracked export command that reads the configured GBrain brain and writes:

- `manifest.json`
  - schema version
  - corpus identifier and content hash
  - GBrain version
  - chunker version
  - embedding model and dimensions
  - page, chunk, edge, and vector counts
  - hashes for every artifact file
- `pages.jsonl`
  - stable page ID
  - source ID
  - slug
  - title
  - perturbation labels
- `chunks.jsonl`
  - stable chunk ID
  - page ID
  - chunk index
  - chunk text
- `embeddings.npy`
  - one OpenAI embedding per chunk in `chunks.jsonl` order
- `links.jsonl`
  - source page ID
  - target page ID
  - link type and provenance when available

Canonical frozen artifacts should be repo-visible under `data/`: retrieval indexes under `data/retrieval-indexes/`, expansion caches under `data/expansion-caches/`, and replayable result artifacts under `data/benchmark-results/`. Private rebuild state, package caches, and live GBrain runtime data should remain under ignored `.gbrain-local/`. Prediction config sidecars must record the retrieval manifest hash so a run can be tied to an exact snapshot.

The loader must reject missing files, hash mismatches, duplicate IDs, dangling page or edge references, vector-count mismatches, unexpected dimensions, and embedding models other than the configured OpenAI model.

## Retrieval Pipeline

### Shared behavior

- Rank chunks internally but return pages.
- A page's semantic score and injected context use the chunk from that page with the highest cosine similarity to the original query.
- Deduplicate by stable page ID.
- Break score ties by page ID for deterministic output.
- Use a per-list candidate depth of 20.
- Use a final top-k of 5 for official experiments.
- Admit all retrieved memories. Perturbation labels are evaluation metadata and must not affect retrieval.
- Do not use autocut, a cross-encoder reranker, compiled-truth boosts, source boosts, recency boosts, or dynamic score thresholds.

### `dense`

1. Embed the original retrieval query with the same OpenAI model and dimensions as the frozen index.
2. Compute cosine similarity against the exported chunk matrix.
3. Keep the best chunk per page.
4. Sort pages by cosine and return the fixed top-k.

Dense mode does not use BM25, RRF, query expansion, title retrieval, or graph signals.

### `hybrid`

1. Produce a body-BM25 page ranking for the original query.
2. Produce an OpenAI-embedding cosine page ranking for the original query.
3. Fuse the two page lists with standard reciprocal rank fusion:

   `RRF(page) = sum(1 / (60 + one_based_rank))`

4. Normalize RRF scores by the largest observed RRF score.
5. Calculate:

   `score = 0.70 * normalized_rrf + 0.30 * original_query_cosine`

6. Sort and return the fixed top-k.

Hybrid mode does not use query expansion, title retrieval, or graph signals.

### `enhanced`

1. Start with the original query.
2. Load exactly three cached alternative queries generated by a separately configured `gpt-5-mini` expansion prompt.
3. For the original and each expansion, produce:
   - a body-BM25 page list
   - an OpenAI-embedding cosine page list
   - a title-only BM25 page list
4. Fuse all twelve lists using the same `K=60` RRF definition.
5. Apply the same 70/30 blend using cosine to the original query, not an expansion.
6. Inspect the leading 20 fused pages as an induced graph:
   - count distinct inbound links from other pages in that pool
   - if a page has at least two such links, multiply its score by `1.05`
   - record the inbound count and factor in the trace
7. Re-sort and return the fixed top-k.

Graph reranking must never introduce a page that was absent from the lexical, dense, and title candidate union. Do not carry over GBrain's cross-source or session-path heuristics.

## Query Expansion Contract

Use a versioned structured-output prompt with `gpt-5-mini` to generate exactly three unique alternatives. Expansion is independent of the answer-generation model.

Cache entries by a hash of:

- normalized original query
- expansion model
- prompt version

Support two policies:

- `populate`: call the expansion model on a cache miss and persist the validated result
- `frozen`: fail on a cache miss and never write

Official benchmark runs use `frozen`. Traces and config sidecars must record the expansion model, prompt version, query hash, alternatives, and cache hash. Invalid, duplicate, or incomplete expansion output is an error rather than a silent fallback.

## Code Architecture

Introduce:

- `RetrievalMode`: `dense`, `hybrid`, or `enhanced`
- `RetrievalRequest`: original query and requested top-k
- `RetrievalHit`: page identity, title, selected chunk, final score, perturbation labels, and score diagnostics
- `RetrievalResult`: ordered hits plus the full retrieval trace
- `Retriever`: interface used by every memory-using condition
- `RetrievalIndex`: validated loader for the frozen GBrain artifact
- `QueryExpander`: cached expansion interface used only by enhanced mode

Refactor baseline execution so a baseline determines the retrieval query and timing, then calls an injected `Retriever`. `InputRagBaseline` should query only with the visible case prompt; later output-RAG and PARM conditions can supply their own query sources without reimplementing ranking.

Replace the canonical `--memory-backend gbrain` path with:

- `--retrieval-mode {dense,hybrid,enhanced}`
- `--retrieval-index PATH`
- `--retrieval-limit`, retained as the shared top-k parameter
- enhanced-only expansion-cache path and policy options

Memory-using baselines must require an explicit retrieval mode and index. `no_memory` must reject retrieval arguments. Do not give memory-using runs an implicit default mode.

## Trace and Configuration Contract

Each prediction must retain the existing baseline and query-source metadata and add:

- retrieval mode
- original query
- expansion queries, when applicable
- ranked candidate IDs from each channel
- per-channel ranks for every returned page
- raw and normalized RRF
- original-query cosine
- pre-graph score
- graph inbound-link count and multiplier
- final score and rank
- selected chunk ID
- retrieved page/source IDs
- perturbation labels

The run config sidecar must additionally record all fixed constants, dependency/model versions, index manifest hash, and expansion-cache hash. This makes every result reconstructable without relying on GBrain's search implementation.

## Implementation Sequence

1. Add and test the frozen GBrain exporter and PARM index validator.
2. Add retrieval data types and the `Retriever` boundary.
3. Implement dense retrieval and refactor `input_rag` to use it.
4. Implement body BM25, fixed RRF, and the 70/30 hybrid score.
5. Implement cached query expansion and title-only BM25.
6. Implement adjacency-only graph reranking.
7. Add CLI validation, prediction traces, and config-sidecar provenance.
8. Update the benchmark documentation and proposal to describe retrieval as an orthogonal axis.
9. Run one Amara case through all three modes, then run the full mode-matched benchmark matrix.

## Tests and Acceptance Criteria

Unit tests must cover:

- OpenAI-embedding cosine ordering and page-level best-chunk selection
- BM25 ordering
- one-based `K=60` RRF math
- RRF normalization and 70/30 scoring
- deterministic tie-breaking
- title retrieval introducing a candidate absent from body retrieval
- page deduplication
- adjacency reranking reordering existing candidates
- graph reranking never introducing unseen pages
- expansion cache population and frozen replay
- malformed, duplicate, or incomplete expansions
- artifact hashes, model identity, dimensions, and referential integrity
- CLI requirements for memory and no-memory conditions

Integration tests should use a tiny deterministic fixture index and a fake expansion cache, requiring no network or live GBrain process.

Before the full experiment:

1. Export the Amara brain.
2. Run one case under `input_rag × dense`, `input_rag × hybrid`, and `input_rag × enhanced`.
3. Inspect the three retrieval traces manually.
4. Repeat each run and confirm identical page ordering and scores within floating-point tolerance.
5. Confirm no canonical run invokes `gbrain search`.

The feature is complete when all retrieval-capable conditions use the shared retrieval interface, the three modes operate from the same frozen index, and official results can be reproduced from their index and expansion-cache hashes.
